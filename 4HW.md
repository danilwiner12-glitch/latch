# Slack


В цифровой схемотехнике **slack** — это запас по времени для сигнала относительно заданного временного ограничения.

Проще говоря, slack показывает, **успевает ли сигнал прийти вовремя**.

- **Положительный slack**: путь укладывается в ограничение, есть временной запас.
- **Нулевой slack**: путь ровно на границе допустимого.
- **Отрицательный slack**: путь нарушает ограничение, сигнал приходит слишком поздно или слишком рано.

Slack — один из ключевых результатов **static timing analysis (STA)**.

Для любого временного пути сравнивают:

- **required time** — момент, к которому сигнал должен прийти;
- **arrival time** — момент, когда сигнал реально приходит.

Тогда:

```text
slack = required time - arrival time
```

Если результат:

- `> 0` — всё хорошо;
- `= 0` — на пределе;
- `< 0` — есть timing violation.


## Виды slack

### 1. Setup slack

**Setup slack** показывает, успевает ли данные дойти до входа триггера **до активного фронта такта** с учетом setup time.

Идея:
- данные стартуют из одного регистра;
- проходят через комбинационную логику;
- должны стабилизироваться на входе следующего регистра заранее.

Упрощённо:

```text
setup slack = required setup time - actual data arrival time
```

Если **setup slack отрицательный**, это означает, что путь слишком длинный, и максимальная частота работы схемы может быть недостижима.


### 2. Hold slack

**Hold slack** показывает, не меняются ли данные **слишком рано** после фронта такта, нарушая hold time принимающего регистра.

Упрощённо:

```text
hold slack = actual data arrival time - required hold boundary
```

Если **hold slack отрицательный**, это означает нарушение hold: данные пришли слишком быстро, и принимающий регистр может захватить неверное значение.

---

## Пример для setup

Пусть:

- период такта: `10 ns`
- задержка запускающего регистра `clk->Q`: `0.8 ns`
- задержка комбинационной логики: `6.5 ns`
- setup time принимающего регистра: `1.2 ns`
- skew и uncertainty для простоты считаем нулевыми

Тогда:

```text
arrival time = 0.8 + 6.5 = 7.3 ns
required time = 10 - 1.2 = 8.8 ns
slack = 8.8 - 7.3 = 1.5 ns
```

Итог: **setup slack = +1.5 ns**, путь корректен и имеет запас.

---

## Пример отрицательного setup slack

Если задержка комбинационной логики выросла до `8.5 ns`, тогда:

```text
arrival time = 0.8 + 8.5 = 9.3 ns
required time = 8.8 ns
slack = 8.8 - 9.3 = -0.5 ns
```

Итог: **setup slack = -0.5 ns**.

Это означает, что путь не успевает к нужному фронту такта. Для исправления обычно:

- уменьшают глубину логики;
- перераспределяют pipeline;
- оптимизируют размещение и трассировку;
- ослабляют timing constraint, если это допустимо;
- снижают рабочую частоту.

---

## Пример для hold

Пусть:

- минимальная задержка `clk->Q`: `0.1 ns`
- минимальная задержка логики: `0.05 ns`
- hold time принимающего регистра: `0.3 ns`

Тогда данные приходят через:

```text
arrival time = 0.1 + 0.05 = 0.15 ns
```

Если требуется, чтобы данные оставались стабильными хотя бы `0.3 ns`, то:

```text
hold slack = 0.15 - 0.3 = -0.15 ns
```

Итог: **hold slack отрицательный**, есть hold violation.

Типичные способы исправления:

- добавить буферы в быстрый путь;
- изменить маршрутизацию;
- скорректировать clock skew;
- использовать специальные hold-fix оптимизации на этапе P&R.

---

## Где slack встречается на практике

Slack анализируют в:

- FPGA-проектах;
- ASIC-дизайне;
- синтезе;
- place-and-route;
- static timing analysis;
- sign-off проверках.

Во временных отчетах обычно встречаются:

- **WNS** — Worst Negative Slack  
  Худший отрицательный slack среди всех путей.

- **TNS** — Total Negative Slack  
  Сумма всех отрицательных slack по нарушенным путям.

- **Setup violations count** / **Hold violations count**  
  Количество путей с нарушениями.

---

## Почему slack может ухудшаться

На slack влияют:

1. **Длинная комбинационная логика**  
   Чем больше уровней логики, тем больше задержка.

2. **Большая длина межсоединений**  
   После placement/routing задержки проводников могут стать доминирующими.

3. **Высокая тактовая частота**  
   Меньший период такта уменьшает required time.

4. **Clock skew и jitter**  
   Неидеальность тактового дерева ухудшает запас.

5. **PVT-углы**  
   Process, Voltage, Temperature сильно влияют на задержки.

6. **Неправильные constraints**  
   Ошибки в SDC/XDC могут создать ложные или пропущенные нарушения.

---

## Как интерпретировать slack

### Если setup slack отрицательный
Это проблема **максимальной задержки** пути.

Возможные последствия:
- схема не держит нужную частоту;
- возникают ошибки выборки данных;
- требуется оптимизация критических путей.

### Если hold slack отрицательный
Это проблема **минимальной задержки** пути.

Возможные последствия:
- регистр захватывает нестабильные данные;
- ошибка может проявляться даже на низких частотах;
- нарушение hold обычно особенно критично и должно исправляться физически.

---

## Критический путь и slack

**Критический путь** — путь с наихудшим slack, обычно с минимальным setup slack.

Именно он чаще всего определяет максимальную достижимую частоту:

```text
Fmax ≈ 1 / Tclk_min
```

где `Tclk_min` выбирается так, чтобы setup slack стал неотрицательным.

---

## Связь slack с STA

В static timing analysis инструмент:

1. строит временные пути;
2. вычисляет задержки элементов и межсоединений;
3. определяет arrival time;
4. вычисляет required time;
5. находит slack для каждого пути.

Это позволяет проверить корректность схемы **без полного динамического моделирования всех возможных состояний**.

---

## Что важно помнить

- **Slack — это запас по времени**, а не просто задержка.
- **Положительный slack** означает выполнение ограничения.
- **Отрицательный slack** означает timing violation.
- Для надежной работы нужно контролировать и **setup**, и **hold**.
- Один и тот же дизайн может иметь разные slack в разных PVT-углах и режимах работы.

---

## Краткое резюме

Slack в цифровой схемотехнике — это численная оценка того, насколько временной путь укладывается в заданное ограничение.

Ключевые идеи:

- `slack = required time - arrival time`
- **setup slack** связан с тем, что данные не должны прийти слишком поздно;
- **hold slack** связан с тем, что данные не должны прийти слишком рано;
- отрицательный slack означает нарушение временных требований;
- анализ slack — центральная часть STA, FPGA и ASIC flow.
